# Study Agent 产品方向

> 状态：正式产品边界
> 更新：2026-07-18

## 1. 产品定义

**Study Agent 是一个能够长期保持“我正在学什么、已经确认什么、还不会什么、下一步是什么”的个人学习工作台。**

所有新增能力都必须先回答一个问题：

> **它是否帮助用户更好地继续学习？**

如果不能直接改善学习连续性、理解验证、资料使用、证据可信度或下一步行动，就不应进入普通用户主路径。

## 2. 能力层级

### GitHub

GitHub 能力不是第二个产品，而是：

> **学习源码时使用的高级研究工具。**

仓库快照、代码结构、Git 历史、PR/CI、change impact、review context 和未来受控执行，都必须回到“帮助用户理解源码并继续当前学习目标”的主链。

### RAG

RAG 不是“知识库产品”，而是：

> **围绕自己的资料学习。**

普通用户首先看到的是“上传资料、围绕资料提问、查看引用”，而不是索引、向量数据库、topK 或内部检索参数。

### Web Research

Web Research 不是“搜索引擎”，而是：

> **需要外部事实时获得可信证据。**

搜索、来源筛选、阅读和 EvidenceTrail 是学习回答的证据基础设施，不应形成与学习工作台平级的主产品。

### Memory

Memory 不是一个工作区，而是：

> **学习连续性基础设施。**

用户关心的是“这次确认了什么、还缺什么、下次从哪里继续”，而不是 memory 文件、run ID 或存储实现。

### Workflow

Workflow 不属于普通用户主功能。

它属于：

> **高级诊断 / 开发者模式。**

普通用户只需要看到当前任务是否正在进行、是否失败、能否继续或重试。完整运行记录和内部阶段用于诊断。

## 3. 普通用户主路径

普通用户主界面优先保持以下结构：

1. 当前学习目标 / 当前任务。
2. 当前已确认内容。
3. 当前未解决缺口。
4. 明确下一步。
5. 对话输入。
6. 按需出现的资料、来源和恢复动作。

低频设置、实验功能和开发者诊断不得与学习主路径争夺一级注意力。

## 4. 近期收敛顺序

在下面三个收敛 PR 完成前，停止无明确学习价值的横向功能扩张。

### PR A：产品边界清理（P0）

- 删除输入框永久 TaskIntent 下拉框，改为轻量的“自动 · 当前任务”Chip；只有用户需要纠正系统判断时才展开。
- Settings 不再渲染 legacy Sidebar，设置页只保留学习体验、资料辅助和隐私相关设置。
- 设置页移除上传、新建会话、工作区导航、工具、工作流和记忆入口。
- More 一级只保留：资料与来源、学习成果、设置。
- 群聊、新闻、工具、工作流归入“实验功能”；工作流以开发者诊断呈现。

### PR B：上传资料到开始学习的完整流程

目标路径：

```text
上传资料
-> 正在解析
-> 资料已准备好
-> 基于这份资料：
   [开始系统学习]
   [直接提问]
```

普通用户不应先理解 upload/rebuild、retrieval mode、topK、vector backend 等内部概念。

“重建整个知识库”只能放在知识管理的危险操作区。

### PR C：产品级流畅度回归

固定五条 Golden Journey：

1. 第一次打开 -> 提问 -> 获得回答。
2. 系统学习 -> 建立目标 -> 学习 -> 理解验证 -> 整理 -> 下次恢复。
3. 上传文档 -> 等待完成 -> 围绕资料提问 -> 查看引用。
4. 联网研究 -> 查看研究进度 -> 中止 -> 恢复 -> 继续对话。
5. GitHub 源码学习 -> 输入仓库 -> 阅读源码 -> 回到学习目标。

验收指标不只检查“按钮存在”，而要记录：

- 首次回答前需要几次决策。
- 输入问题前有几个控件要求理解。
- 一个任务跨越多少个页面或抽屉。
- 是否存在重复入口。
- 是否暴露内部术语。
- 中断后几步能够恢复。
- 用户是否始终知道下一步做什么。

## 5. 新功能准入门禁

新增普通用户功能前必须满足至少一项：

- 帮助建立或恢复学习目标。
- 帮助验证理解，而不是制造虚假的掌握感。
- 帮助使用自己的资料学习。
- 帮助获得并核对外部证据。
- 帮助明确当前缺口和下一步。
- 显著减少完成上述任务的操作成本。

否则默认进入实验功能、开发者诊断，或暂不进入产品主线。
